   - 既存アニメーションや UI 命名規則
3. その結果を踏まえて、最も自然な組み込み方法を選ぶ

実装したい演出仕様:
1. ローディング全体
- 演出時間は合計 2.0 秒
- 開始時にオーバーレイを 0.3 秒でフェードイン
- 0.5 秒時点で 16%
- 1.0 秒時点で 50%
- 1.5 秒時点で 99%
- 1.5 秒〜2.0 秒でオーバーレイをフェードアウト
- フェードアウト中に、遷移先ページが後ろから少し浮かび上がるように見せる
- ローディングバーは演出用でよく、実際の通信進捗とは連動しなくてよい

2. 背景演出
- 背景の土台画像は使わない
- 2進数は上から下へ流す
- 16進数は右から左へ流す
- サイバー / ニューロン風の雰囲気
- 実装方式は canvas / CSS / SVG のいずれでもよいが、軽量であること
- 不要な画像アセットは増やさないこと

3. 遷移仕様
- 対象は通常のページ遷移リンク（aタグ）
- JavaScript が有効な場合はクリックをフックして演出を実行
- JavaScript 無効時は通常のリンク遷移がそのまま動くこと
- 同一ページ内アンカー、mailto、tel、外部リンク、新規タブ、ダウンロードリンクなどは必要に応じて除外すること
- 二重クリック対策を入れること

4. 「本当に次ページが浮かび上がる」見せ方
- 可能なら、遷移先ページを先読みして、実際の遷移先 HTML を使って背後表示に近い表現を実装すること
- もしブラウザ仕様やサイト構成上それが難しい場合は、できるだけ自然な代替表現にすること
- その場合は、README かソースコメントで
  - 本当に先読みしているのか
  - 演出的にそう見せているのか
  を明記すること

5. キャラクターについて
- 最終的には、左から右へ走るアニメキャラクター風の女性をこの演出に追加したい
- ただし今回はまず基盤実装を優先する
- 実装は「後からキャラを差し込める構造」にしておくこと
- 具体的には以下を考慮する
  - キャラ用レイヤーを分離する
  - sprite sheet または連番画像を差し込めるようにする
  - キャラなしでも破綻しない設計にする
- もし適切なら、ダミー要素または TODO コメントで追加ポイントを明示すること

6. 品質要件
- グローバル汚染をなるべく避ける
- モバイルでも破綻しない
- prefers-reduced-motion に配慮する
- 既存スタイルを壊さない
- 複数ページで再利用可能にする

7. 成果物

- どのファイルを編集したか

- README またはコメントによる簡単な導入説明

8. 検証
- 実際にローカルで起動して動作確認すること
- 少なくとも主要なページ遷移リンクで確認すること
- 以下を確認すること
  - 2進数が上から下へ流れる
  - 16進数が右から左へ流れる
  - 0.5s, 1.0s, 1.5s で 16%, 50%, 99% になる
  - 1.5s〜2.0s でフェードアウトしながら次ページが浮かび上がる見え方になる
  - JavaScript 無効時に通常遷移できる
  - 二重クリックで壊れない
  - モバイル幅でも破綻しない
